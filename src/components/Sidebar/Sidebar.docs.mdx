# Sidebar

A collapsible navigation rail. `Sidebar` owns the collapsed/expanded state
and provides it via context; `SidebarItem` reads that context to know
whether to hide its label, badge, and expand chevron — items don't manage
the collapsed state themselves.

## Collapsed state

Uncontrolled usage passes `defaultCollapsed` and `Sidebar` tracks the
collapsed/expanded state internally, toggled by the built-in header button.
Controlled usage passes both `collapsed` and `onCollapsedChange`, and the
consumer owns the state entirely — same shape as an uncontrolled/controlled
form input.

```tsx
const [collapsed, setCollapsed] = useState(false);

<Sidebar collapsed={collapsed} onCollapsedChange={setCollapsed}>
  <SidebarItem icon={<Home className="size-4" />} label="Home" active />
</Sidebar>
```

## Nested items

`SidebarItem` accepts nested `SidebarItem` children, rendered as a
collapsible group below it and toggled open on click (or expanded by
default when `active` is set). Nesting is skipped visually while the
sidebar itself is collapsed.

## Links vs. buttons

`SidebarItem` renders an `<a>` when `href` is provided, or a `<button>`
otherwise — pick whichever matches whether the item navigates to a route or
just triggers a click handler.

## Overriding styles

Both `Sidebar` and `SidebarItem` resolve their `className` prop through the
same `cn()` merge utility the rest of the library uses, so an app-level
override always wins over the component defaults:

```tsx
<Sidebar className="border-brand-500 shadow-z3">
  <SidebarItem icon={<Home className="size-4" />} label="Home" />
</Sidebar>
```

## Composition

`Sidebar` forwards its ref to the root `<aside>` and spreads all native
`<aside>` attributes. `SidebarItem` forwards its ref to whichever element it
renders (`<a>` when `href` is set, `<button>` otherwise) and spreads all
remaining native attributes for that element, so both compose with the rest
of the library without a separate "as child" API. `SidebarItem` falls back
to a no-op collapsed state when rendered outside a `Sidebar`, rather than
throwing, since standalone usage (e.g. in isolated tests) is reasonable.
