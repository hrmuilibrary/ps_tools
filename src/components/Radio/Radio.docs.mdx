# Radio

A labeled radio input. The native `<input type="radio">` is visually hidden
(`sr-only`) but stays in the tab order and drives every visual state of a
custom dot via the `peer` variant, so the component keeps full native
keyboard/form/a11y behavior while looking however the design wants.

## RadioGroup

`Radio` is almost always used inside a `RadioGroup`, which renders a
`role="radiogroup"` container and provides `name`, `size`, and `disabled` to
every `Radio` descendant via context — so those don't need to be repeated on
each `Radio`. The group also owns the controlled/uncontrolled selection:
pass `value`/`onChange` for a controlled group, or `defaultValue` for an
uncontrolled one, and compare against each `Radio`'s own `value` prop.

```tsx
const [value, setValue] = useState('comfortable');

<RadioGroup name="density" value={value} onChange={setValue}>
  <Radio value="compact" label="Compact" />
  <Radio value="comfortable" label="Comfortable" />
  <Radio value="spacious" label="Spacious" />
</RadioGroup>;
```

`orientation` (`'vertical'` default, or `'horizontal'`) controls the layout
of the group's children. A `Radio` rendered outside any `RadioGroup` falls
back to its own `checked`/`defaultChecked`/`onChange` props and behaves like
a standalone controlled/uncontrolled input.

## Error state

`error` swaps the border (and the ring color stays brand) to the danger
palette and sets `aria-invalid` on the input, for use alongside form
validation messaging rendered by the consumer.

```tsx
<Radio value="option" error label="This option is currently unavailable" />
```

## Composition

`Radio` forwards its ref to the native `<input>` and spreads all native input
props (minus `size`/`type`/`name`/`onChange`, which the component or its
`RadioGroup` own), so it composes with forms and validation libraries without
a separate API.
